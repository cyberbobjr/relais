# Commandant — Slash Command Dispatcher Brick

## Functional Overview

Commandant is the slash-command processor of the RELAIS pipeline. When a user prefixes a message with `/`, Sentinelle identifies it as a command, validates it against the ACL, and routes it to Commandant via a dedicated stream. Commandant then parses the command and dispatches it to the appropriate handler.

Handlers vary in whether they produce a direct reply: `/help` and `/bundle` sub-commands reply immediately to the originating channel; `/clear`, `/sessions`, and `/compact` delegate all output to downstream bricks (Souvenir and Atelier); `/resume` replies only on error. All commands acknowledge unconditionally.

It operates as a pure synchronous dispatcher: no LLM calls, no SQLite, no per-message state.

---

## Architecture Position in the Pipeline

```
Sentinelle (incoming)
  │
  ├── relais:commands ──────────────────► Commandant (commandant_group)
  │                                         │
  │                                         ├── relais:messages:outgoing:{channel}   (reply to user)
  │                                         ├── relais:memory:request                (clear / sessions / resume)
  │                                         ├── relais:atelier:control               (/compact)
  │                                         └── relais:tasks                         (/bundle install setup)
```

Commandant sits outside the main Atelier loop — commands are low-latency administrative operations that must not compete with LLM tasks for concurrency. It uses `ack_mode="always"`: an unhandled exception in one command handler must never block subsequent commands.

---

## Command Dispatch Flow

Every envelope arriving on `relais:commands` goes through the same three-phase processing loop:

### Phase 1 — Parse

`parse_command(envelope)` splits `envelope.content` into a command name (e.g. `clear`) and a list of arguments. The command name is looked up in `COMMAND_REGISTRY`. If the name is unknown, a user-visible error reply is returned immediately.

### Phase 2 — Dispatch

The handler function registered in `COMMAND_REGISTRY` receives the envelope and the parsed arguments. Handlers are plain `async def` functions; they may publish multiple envelopes to different output streams.

### Phase 3 — ACK

The envelope is always acknowledged regardless of handler outcome. If a handler raises, the exception is logged and the ACK still occurs.

## Command Registry

`COMMAND_REGISTRY` (defined in `commandant/commands.py`) maps each command name to its handler function.

| Command | Handler | Outputs |
|---|---|---|
| `/clear` | `handle_clear` | `relais:memory:request` (clear action) |
| `/help` | `handle_help` | `relais:messages:outgoing:{channel}` (dynamic list from `COMMAND_REGISTRY`) |
| `/sessions` | `handle_sessions` | `relais:memory:request` (sessions action) |
| `/resume` | `handle_resume` | `relais:memory:request` (resume action); error reply to channel if no session_id |
| `/compact` | `handle_compact` | `relais:atelier:control` (op="compact") |
| `/bundle` | `handle_bundle` | Delegated to `commandant/bundle_commands.py` (see below) |

### `/clear`

Publishes an envelope to `relais:memory:request` with `action=ACTION_MEMORY_CLEAR` and the user's `session_id`. Souvenir drops the corresponding Redis context key and deletes all short-term memory entries for the session. Commandant sends no confirmation reply; any feedback to the user is generated by Souvenir's handler or a subsequent pipeline step.

### `/help`

Dynamically builds the list of available commands from `COMMAND_REGISTRY.values()`, formatting each entry as `/<name> — <description>`. Publishes the result directly to `relais:messages:outgoing:{channel}`. No external calls. The list is always up-to-date because it is generated from the live registry, not hardcoded.

### `/sessions`

Publishes to `relais:memory:request` with `action=ACTION_MEMORY_SESSIONS` and the user's `user_id`. Commandant does not BRPOP the response — Souvenir handles producing the session list asynchronously to the appropriate output stream.

### `/resume`

Validates that a session ID argument was provided; if missing, sends an error reply directly to the originating channel. If a session ID is present, publishes to `relais:memory:request` with `action=ACTION_MEMORY_RESUME` and the target `session_id`. Souvenir reloads the requested session into the active Redis context key so Atelier picks up the restored history on the next user message. Commandant sends no confirmation reply after a successful request.

### `/compact`

Publishes a control envelope to `relais:atelier:control` with `op="compact"` and the current `session_id`. Atelier's control consumer receives it, runs `AgentExecutor.compact_session()`, and summarises the conversation history, reducing checkpoint size. Commandant sends no reply to the user; any confirmation is produced by Atelier.

## Bundle Sub-Commands

`handle_bundle` in `commandant/commands.py` delegates to `commandant/bundle_commands.py`, which dispatches on the first argument (`install`, `uninstall`, `list`):

### `/bundle install <path>`

```
1. Validate <path> argument is present
2. Call common.bundles.install_bundle(path)
3. Reply to user: success / error message
4. If success: publish trigger envelope → relais:tasks
   with force_subagent="relais-config" to perform post-install setup
```

The trigger envelope follows the virtual channel pattern: `sender_id` and `session_id` are inherited from the original user envelope, so post-install output is routed back to the correct channel.

### `/bundle uninstall <name>`

```
1. Validate <name> argument is present
2. Call common.bundles.uninstall_bundle(name)
3. Reply to user: success / error message
```

Hot-reload in Atelier picks up the removed directory automatically via watchfiles.

### `/bundle list`

```
1. Call common.bundles.list_bundles()
2. Format the list (name, version, description per bundle)
3. Reply to user
```

`_send_reply(envelope, text)` is the shared helper used by all bundle sub-handlers. It creates a `create_response_to(envelope)` envelope, sets `action=ACTION_MESSAGE_OUTGOING`, and publishes to `relais:messages:outgoing:{channel}`.

---

## Data Model

Commandant holds no SQLite database and sets no Redis keys. All state it needs comes from the incoming envelope (session_id, sender_id, channel, context). Persistent effects (clearing memory, restoring a session) are delegated to Souvenir via `relais:memory:request`.

---

## Redis Keys

Commandant does not read or write any Redis keys beyond publishing to streams.

| Stream | Direction | Purpose |
|---|---|---|
| `relais:commands` | Consume | Incoming slash-command envelopes from Sentinelle |
| `relais:messages:outgoing:{channel}` | Produce | User-facing replies |
| `relais:memory:request` | Produce | Memory operations (clear, sessions, resume) |
| `relais:atelier:control` | Produce | Compact operation forwarded to Atelier |
| `relais:tasks` | Produce | Bundle post-install setup tasks |

---

## Configuration Reference

Commandant has no dedicated YAML configuration file. All behaviour is hard-coded in the command registry. The only externally tunable aspect is the Redis connection shared via the common `config.yaml`.

## Important Design Decisions

### Why `ack_mode="always"`?

Commands are administrative: a failed `/clear` or `/help` is not a reason to leave the envelope in the PEL indefinitely. Commandant's handlers are stateless and idempotent for most operations, so re-delivery would at worst produce a duplicate reply. Unconditional ACK avoids PEL accumulation and keeps the command channel responsive.

### Why no LLM in Commandant?

Slash commands require deterministic, low-latency responses. Introducing an LLM call would add variable latency and failure modes. The `/compact` command is the only one that triggers an LLM call — but that call happens inside Atelier's dedicated executor, isolated from Commandant's processing loop.

### Why delegate `/compact` to `relais:atelier:control`?

The `AsyncSqliteSaver` checkpointer managing LangGraph conversation history is owned by Atelier's lifespan context. Commandant has no access to it. By publishing a control envelope, Commandant offloads the checkpoint manipulation to the only process that legitimately owns it.

### Why forward `/bundle install` to `relais:tasks`?

Some bundles include subagent `system_prompt` files that reference configuration steps. Routing the post-install setup through Atelier (via a `force_subagent` task) ensures the setup is executed with the full tool set (file I/O, config CRUD via `relais-config` subagent) rather than requiring Commandant to implement custom post-install logic per bundle.
